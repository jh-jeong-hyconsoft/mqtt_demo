# MQTT Server Client Design

## Goal

Build a local development MQTT setup where the broker runs in Docker and the server client runs from a Python virtual environment.

## Confirmed Facts

- `mqtt_broker/` had no Docker or Mosquitto configuration.
- `mqtt_client/config/topic_config.json` defines one inbound server publish topic: `server/{fleet_id}/{device_id}/command`.
- The robot-side topics `state`, `presence`, `event`, and `ack` are outbound from the robot perspective, so the server client should subscribe to them for logging.
- `mqtt_client/config/schema_v0.3_validation.json` requires full message envelopes, not payload-only JSON.

## Design

The server client exposes two CLI subcommands:

- `log`: subscribes to robot topics and writes every received MQTT message to console plus a JSON Lines log file.
- `publish`: reads a payload-only JSON file, wraps it as a `command` message, validates the full envelope against the schema, and publishes it to the configured command topic.

The command envelope is generated by the client. It fills `msg_schema_version`, `msg_id`, `msg_ts_ms`, `msg_type`, `device_id`, `fleet_id`, `connectivity`, `session_id`, and `msg_payload`.

## Error Handling

- Missing or invalid payload JSON exits before publishing.
- Schema validation failures exit before publishing.
- MQTT connection failures surface as CLI errors.
- Log mode stores raw payload text and parsed JSON when parsing succeeds.

## Testing

Unit tests cover topic selection, envelope generation, schema validation, and log record formatting. MQTT network behavior remains manually verifiable against the Docker broker.
